# HTML 转 Markdown 工具细节

这份文档讲三件事：

1. 为什么这次要做“Markdown + 少量图片”，而不是“整页截图”。
2. 代码内部是怎么分工的。
3. 当前脚本有哪些边界。

## 1. 为什么不能再全做成 PNG

你这次的目标和之前不一样。

之前整页导出 PNG，适合：

- 快速发图
- 保留页面视觉设计

但它也有明显问题：

- 手机端看字还是容易小
- 正文无法复制
- 知乎里二次编辑很难
- 一整页都是图，平台阅读体验差

所以这次的策略变成：

- **正文转 Markdown**
- **流程图 / 结构图 / 验证图保留为 PNG**

这正好适合知乎。

## 2. 目录里的脚本怎么分工

### 2.1 `export_html_to_markdown.py`

这是主脚本。

它负责两件事：

1. 把 HTML 主体转成 Markdown。
2. 如果开了 `--capture-images`，先调用抽图脚本，把带 `svg` 的图块导成 PNG，再把图片路径插进 Markdown。

它现在的做法不是暴力把整个 HTML 原样翻译，而是按你这页的结构做半自动转换：

- `hero` 变成标题 + 一句话版 + 关键数字
- `section.card` 变成 `##` 章节
- `grid2 / grid3 / grid4` 里的 `item` 变成小标题 + 正文
- `div.box` 里带 `svg` 的内容变成“标题 + 图片 + note”
- `div.box` 里不带 `svg` 的内容继续转成文字

也就是说，它默认站在“保留阅读逻辑”的角度，不是“保留 HTML 排版”的角度。

### 2.2 `capture_svg_boxes.js`

这是“只抽图”的脚本。

它会：

1. 打开本地 HTML。
2. 用 Chrome DevTools 协议读取页面。
3. 找出所有 `div.box` 中带 `svg` 的块。
4. 只截这些图块，不截整页。
5. 生成一个 `diagram_manifest.json`，记录“标题 -> 文件名”的映射。

这样主脚本就能知道：

- `一轮 RL step 是怎么跑的` 对应哪张图
- `SD 在做什么` 对应哪张图
- `TLT 总体结构` 对应哪张图

然后 Markdown 里自动插图。

### 2.3 `export_html_to_png.js / ps1`

这是你之前那套整页导图能力，我按你的要求一起放进来了。

它适合：

- 想继续导整页 PNG
- 想出知乎手机友好版整页切图
- 想再压成 `1080w` JPG

这套能力并没有丢，只是现在在这个目录里属于“附带工具”，不是主流程。

## 3. Markdown 转换逻辑

当前 `export_html_to_markdown.py` 的主要规则是：

### 3.1 Hero 区

- `h1` -> `#`
- `lede` -> 开头导语
- `chips` -> 列表
- `hero-box` -> 引述块
- `metric-grid` -> 关键数字列表

### 3.2 Card 区

- `section-head h2` -> `##`
- `section-head p` -> 章节导语

### 3.3 Item 卡片

每个 `item` 变成：

- 一个较低层级标题
- 一段正文
- 如果有 `tag`，则转成说明性 note

### 3.4 带 SVG 的 Box

这类 box 会被处理成：

1. 标题
2. 一张 PNG
3. box 里原有的 note 继续转成文字

所以最终效果是：

- 图本身保真
- 图下解释依然可编辑

### 3.5 Note

当前统一转成 Markdown 引述块：

```md
> 这里是说明
```

## 4. 为什么抽图脚本只截 `svg box`

因为这正是“最值得保真”的部分。

如果把普通文字卡片也截图，会立刻回到老问题：

- 文字不可编辑
- 手机端不友好

所以现在只保留这些图块为图片：

- 流程图
- 对比图
- 验证图
- 总体结构图

而下面这些尽量留成文字：

- 概念解释
- 优缺点判断
- Design A / B 的文字分析

## 5. 当前脚本的边界

这套工具不是“任何 HTML 都能完美转 Markdown”的通用工具。

它当前依赖下面这些假设：

- 页面结构比较规整
- 主体内容在 `section.hero` 和 `section.card`
- 图块在 `div.box > svg`
- 卡片文字主要是 `item / box / note`

所以它最适合你这类“单文件论文解读页”。

如果以后你换成很复杂的网页结构，比如：

- 深层嵌套很多
- 图文布局极不规则
- 大量脚本动态生成 DOM

那就需要再调整规则。

## 6. 推荐工作流

以后你如果要从 HTML 发知乎，建议按这个顺序：

1. 优先跑 `publish_html_to_zhihu.ps1 -CopyToClipboard`
2. 直接去知乎编辑器粘贴
3. 如果需要公网可访问图片，再加 `-PushPages`
4. 只有平台不接受富文本粘贴时，再退回使用 `public.md`

也就是说，现在默认路线已经不是“先拿 md 再手工补图”，而是“先拿可直接粘贴的富文本 HTML，md 作为兜底”。

## 7. 对典型论文解读页会得到什么

对一篇典型的论文解读单页 HTML，理想输出就是：

- 绝大多数解释保留成文字
- 这些图被抽出来：
  - 流程图
  - 对比图
  - 多位置验证图
  - 结构图
  - 总体框架图

这正符合这套工具的设计目标：

> 不要把绝大多数正文都做成图片，图片只保留给最值得保真的图块。
